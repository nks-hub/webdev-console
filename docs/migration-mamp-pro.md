# Migrating from MAMP PRO

NKS WebDev Console ships with a built-in `MampMigrator` that reads MAMP PRO's
SQLite configuration and recreates your sites + databases inside NKS WDC.

## What gets migrated

| MAMP PRO data                              | Where it lands in NKS WDC                         |
| ------------------------------------------ | ------------------------------------------------- |
| Virtual hosts (`VirtualHosts` table)       | `~/.wdc/sites/<domain>.toml` — full SiteConfig    |
| Server aliases (`VirtualHostServerAlias`)  | Added as `aliases = [...]` on each site           |
| Project document roots                     | Linked into the new site (no copy)                |
| Per-host PHP version                       | Mapped to NKS WDC's installed PHP versions        |
| MySQL databases                            | Dumped from MAMP MySQL, re-loaded into NKS MySQL  |
| User passwords                             | Re-imported as-is (auth metadata preserved)       |

## What does **not** migrate automatically

- MAMP PRO's `mod_perl`, `mod_python`, third-party Apache modules — you have
  to install them separately on NKS WDC if you actually use them.
- Custom MAMP PRO Apache configs you edited by hand outside the UI — review
  `~/.wdc/sites/<domain>.toml` after migration and tweak any extension/INI
  overrides.
- macOS Keychain entries from MAMP PRO. NKS WDC stores its MySQL root
  password in macOS Keychain under its own service name.

## Before you start

1. **Stop MAMP PRO completely.** It binds to ports 80/443/3306/3308 and will
   collide with NKS WDC's services. (`Cmd-Q` is not enough — make sure no
   `httpd`/`mysqld` from MAMP is still running: `pgrep -fla 'MAMP|mysqld'`.)
2. **Note your MAMP MySQL port.** MAMP PRO defaults to **3308** (not 3306) so
   it doesn't conflict with system MySQL. If you changed it, you'll need that
   number for the migrator.
3. **Back up `~/Library/Application Support/Appsolute/MAMPPRO/`** (mac) or
   `%APPDATA%\Appsolute\MAMPPRO\` (Win) before any operation. It's small and
   the migrator only reads it, but you want a rollback path.

## Run the migrator

```bash
# Dry-run — print what would happen without changing anything
wdc migrate mamp --port 3308

# Actually do it
wdc migrate mamp --port 3308 --apply
```

Verified migration on a real MAMP PRO install:
**17 sites + 66 MySQL databases** moved successfully in under 5 minutes.

The migrator:

1. Reads `mamp.db` (SQLite) — `VirtualHosts` and `VirtualHostServerAlias`
   tables.
2. For each vhost, calls `SiteOrchestrator.CreateSiteAsync` exactly the same
   way the GUI's "New Site" wizard does — vhost is rendered through Scriban,
   SSL cert is generated by mkcert, hosts file gets the entry.
3. Connects to MAMP MySQL on the supplied port, dumps each database with
   `mysqldump`, reloads into NKS WDC's MySQL with `mysql < dump.sql`.
4. Writes a migration report to `~/.wdc/logs/migration-<timestamp>.log`.

Use `--include-domain <pattern>` / `--exclude-domain <pattern>` (glob) to
migrate only a subset.

## After the migration

1. **Browse to a few sites** — `https://<your-old-domain>` should work
   immediately. If a domain doesn't resolve, run `wdc dns sync`.
2. **Run `wdc doctor`** — it checks the obvious things (port collisions,
   PHP extensions, missing services, mkcert CA installed in browsers).
3. **Compare PHP versions** — MAMP PRO ships PHP 5.6 through 8.x. NKS WDC
   uses whatever you've installed via `wdc binaries install php <version>`.
   If a site needs a version that isn't installed yet, install it and the
   site picks it up on next reload.

## Rolling back

The migrator does not modify anything inside MAMP PRO. To roll back:

1. Stop NKS WDC.
2. Start MAMP PRO again — your old sites are intact.
3. Delete the imported sites in NKS WDC (`wdc site delete <domain>` per
   site, or wipe `~/.wdc/sites/`).

## Troubleshooting

If the migration crashes mid-way, the partial result is left in place — fix
the issue (usually a port collision with MAMP still running, or a PHP
version that doesn't exist yet on NKS WDC), then re-run with `--apply`. The
migrator is idempotent: existing sites are skipped.

For other issues see [`troubleshooting.md`](./troubleshooting.md).
